# SquadGo mobile — React Native 0.76.0 → 0.85.3 migration design

**Date:** 2026-06-08
**Status:** Approved (design)
**Goal:** Move the SquadGo mobile app from RN 0.76.0 to 0.85.3 to stay Google-Play target-API compliant and avoid a forced re-migration soon, via a fresh-project rebuild that preserves app identity and source.

## Why (business driver)

Google Play enforces a minimum **target API level** that ratchets up yearly; apps below it stop being accepted for updates. RN 0.76 targets SDK 34 (already behind). RN 0.85 targets a current SDK, restoring compliance and buying a long runway. Doing the jump now avoids another disruptive upgrade in months.

## Approach (locked decisions)

1. **Fresh-project rebuild**, not the upgrade-helper (0.76→0.85 is 9 versions; a clean 0.85 scaffold is more reliable than a giant helper diff).
2. **New Architecture ON** (already enabled at 0.76 — libraries already work under it; keeps us on RN's future default, no second migration).
3. **Brand-new git repo** for the 0.85.3 project. Current 0.76 code is already pushed → old repo is the rollback/archive and the source for `src/`.
4. **Same app identity:** Android `applicationId` + `namespace` = `com.squadgoo.au`; iOS bundle id matched; **reuse existing upload keystore**. Same app on the stores.
5. **Vetted-latest dependency install (Approach B):** install third-party libs at the latest version *confirmed* compatible with RN 0.85 + New Arch + React 19, pinned deterministically. Do NOT touch scaffold core pins. Replace unmaintained libs.
6. **Only the mobile app migrates.** Backend repo untouched; staged Stripe backend work unaffected.

## Section 1 — Strategy & app identity

- Generate fresh **RN 0.85.3** project in a temp dir via the official initializer; move into a new git repo.
- Preserve identity: `applicationId`/`namespace` `com.squadgoo.au`, iOS bundle id, reuse upload keystore + signing config.
- New Arch ON, Hermes ON.
- Copy `src/` + assets from the **current working tree** (includes staged Stripe + job-offer work) into the fresh project.
- Old 0.76 repo (already pushed) = rollback/archive.

## Section 2 — Dependency strategy (vetted-latest, future-proof)

- **Scaffold core pins untouched:** `react@19.2.x`, `react-native@0.85.3`, `@react-native/*`. Never overridden (prevents core-version mismatch — the blanket-install failure mode).
- **Compatibility matrix** (produced in the plan): for each third-party lib, the latest version confirmed to support RN 0.85 + New Arch + React 19, pinned deterministically in `package.json` (reproducible builds — protects against future surprises).
- **Replace unmaintained libs (future-issue insurance):**
  - `react-native-fast-image` (unmaintained) → `@d11/react-native-fast-image` (maintained drop-in fork).
  - `react-native-fs` (stale) → `react-native-blob-util` (maintained); requires a small change to the CSV/PDF export in `src/screens/main/Recruiter/Wallet.jsx`.
  - `@react-native-community/blur`, `react-native-image-crop-picker`, `react-native-keyboard-aware-scroll-view` — verify; replace only if they lack 0.85/new-arch support.
- **Drop junk deps:** `react-native-codegen`, `yarn` (mistakenly in `dependencies`).
- Pure-JS libs (axios, lodash, moment, pdf-lib, yup, react-hook-form, @reduxjs/toolkit, react-redux, @tanstack/react-query, redux-persist) → latest, low risk.
- `react-native-vector-icons` v10+ may have restructured into scoped packages — matrix confirms the correct package/import form (font-linking implications).
- Carry over `@stripe/stripe-react-native` at the latest 0.85-compatible version (it was just added).

## Section 3 — Native config carry-over checklist

**Android:** `applicationId`/`namespace`; signing config + keystore (`MYAPP_UPLOAD_STORE_FILE/KEY_ALIAS/STORE_PASSWORD/KEY_PASSWORD`); `AndroidManifest.xml` permissions; ProGuard/R8 rules; app icons + splash; deep-link intent filters; any custom `MainActivity`/`MainApplication` code; current `targetSdkVersion` (Play-compliant).
**iOS:** bundle id; `Info.plist` permission usage strings (camera/photos/location for image-picker, crop-picker, geolocation, permissions); entitlements; app icons + launch screen; URL schemes; Podfile customizations (Stripe, react-native-permissions `setup_permissions`).
**JS/build:** `babel.config.js` (`@/` alias via `babel-plugin-module-resolver` **+ reanimated plugin LAST**); `metro.config.js`; `react-native.config.js` (vector-icons + custom font asset linking); `tsconfig`/`jsconfig` path aliases; eslint/prettier config; custom fonts/assets.
**Firebase:** none detected (no firebase deps; FCM code commented out in `App.tsx`) — nothing to carry unless confirmed otherwise.

## Section 4 — Build & verification

- Green build on **both** Android (emulator) and iOS (simulator).
- Smoke-test critical flows: auth (login/register/refresh), profile, manual job/offer loop, wallet/Stripe top-up screen render, drawer/tab/stack navigation.
- 8 GB RAM reality: builds are heavy — close other apps; tune Gradle JVM heap modestly within system limits.

## Section 5 — Cutover & rollback

- New repo becomes the project once green + smoke-tested.
- Old 0.76 repo retained as rollback.
- Backend repo untouched (mobile-only migration); staged Stripe backend work unaffected.

## Risks & honest caveats

- **Multi-day effort** with real per-library work; the payoff is Play-Store compliance and no forced near-term re-migration.
- **No absolute "zero future issues" guarantee** — a native upgrade can surface a runtime edge case visible only at testing; Section 4's smoke test is the safety net. Mitigations: maintained libs, deterministic pins, New Arch alignment.
- **8 GB RAM** remains a build-performance/stability constraint regardless of RN version (the recurring local build error may be memory-related and is not necessarily *fixed* by the upgrade).
- Replacing `react-native-fs`/`react-native-fast-image` touches a small amount of app code (export functions, image imports) — covered in the plan.

## Out of scope

- Backend changes; the Stripe/job-offer backend work (separate repo).
- Feature changes — this is an infrastructure upgrade; UI/flows stay as-is.
- Fixing the 8 GB RAM constraint (hardware/environment, not a code task).
